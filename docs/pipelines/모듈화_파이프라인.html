BibleClip · Refactoring Plan

코드 모듈화 파이프라인

3,764줄 단일 파일 autobible.py 를 기능별 패키지로 분할합니다. 동작·기능은 100% 동일, 위험은 단계별 검증으로 0에 수렴.

현재 1개 파일 · 3,764줄 목표 18개 모듈 · 평균 ~210줄 v1.5.0 → v1.6.0

1왜 나누는가

한 파일이 너무 커서, 작은 수정에도 파일 전체가 컨텍스트에 올라가 토큰이 과소비됩니다.

지금의 문제

  • 편집 비용 — 버튼 색 하나 바꿔도 16만 자(≈3,764줄) 전체를 읽어야 함
  • 탐색 비용 — 검색 로직 위치 찾기, 테마 키 확인 등 매번 풀스캔
  • 충돌 위험 — 한 파일에 모든 게 있어 작은 실수가 전체에 영향
  • 거대 클래스 — BibleClipApp 한 클래스가 2,600줄(70%)

나눈 뒤

  • 국소 편집 — 검색 고치려면 search.py(~90줄)만 열면 됨
  • 토큰 ↓ — 작업당 읽는 양이 평균 5~15%로 감소
  • 명확한 경계 — 데이터/UI/업데이트/테마가 물리적으로 분리
  • 안전 — 모듈 단위 테스트·롤백 가능

4거대 클래스 → 믹스인 분할

2,600줄 BibleClipApp를 관심사별 믹스인 7개로. self. 공유 상태는 그대로, 메서드만 파일 이동.

# ui/app.py — 조립만 담당
from bibleclip.ui.viewer    import ViewerMixin
from bibleclip.ui.settings  import SettingsMixin
from bibleclip.ui.lexicon   import LexiconMixin
from bibleclip.ui.search    import SearchMixin
from bibleclip.ui.monitor   import MonitorMixin
from bibleclip.ui.theming   import ThemeMixin
from bibleclip.ui.updater_ui import UpdateMixin

class BibleClipApp(ViewerMixin, SettingsMixin, LexiconMixin,
                   SearchMixin, MonitorMixin, ThemeMixin, UpdateMixin):
    def __init__(self, root):
        ...  # 상태 초기화 + _build_ui() 호출 (믹스인 메서드들이 채워줌)

믹스인은 다중상속으로 한 클래스에 합쳐지므로 self.viewer_text, self.settings 등 공유 상태 접근 방식이 전혀 바뀌지 않습니다. 순수한 "메서드 이사"입니다.

6실행 단계 (검증 게이트 포함)

각 단계 끝에 ✓ 실행 확인 게이트. 통과 못 하면 다음으로 넘어가지 않습니다.

  • Phase 0 · 안전망

    브랜치 + 베이스라인 확보

    git switch -c refactor/modularize 로 작업 브랜치 생성. 현재 v1.5.0이 정상 실행되는지 한 번 띄워 기준점으로 삼습니다. (릴리스 태그는 건드리지 않음 → CI 자동 빌드 트리거 안 됨)

    ✓ v1.5.0 정상 실행 확인
  • Phase 1 · 잎부터 (의존성 없는 순수 모듈)

    config · constants · theme · text_utils 분리

    다른 코드에 의존하지 않는 "잎(leaf)" 모듈을 먼저 잘라냄. 원본 autobible.py 상단을 from bibleclip.config import * 등으로 교체하고 나머지는 그대로 둠.

    ✓ 실행 + 본문 표시 + 다크모드 토글 정상
  • Phase 2 · 데이터·코어 계층

    data/ · core/ (BibleDB·원어·Engine·Formatter)

    UI가 의존하는 백엔드 클래스 이동. import 한 줄만 바뀌고 호출부는 동일. resolve_data_dir 경로 로직이 config.py로 옮겨졌으니 데이터 로딩을 반드시 재확인.

    ✓ 성경 로드 + 원어 패널 + #검색 정상
  • Phase 3 · UI 믹스인 분할 (핵심)

    ui/ 7개 믹스인 + app.py 조립

    한 번에 하나씩 — Theme → Monitor → Search → Settings → Lexicon → Viewer → Update 순(작은 것부터). 각 믹스인 추출 직후 실행해 해당 기능을 직접 눌러봄.

    ✓ 믹스인 1개 추출 = 실행 1회 (총 7회 검증)
  • Phase 4 · 엔트리포인트 정리

    autobible.py를 슈팅으로, __main__.py 추가

    autobible.py 내용을 from bibleclip.ui.app import main; main() 3줄로 축소. PyInstaller spec / CI / build_mac.sh는 여전히 autobible.py를 가리키므로 그대로 동작 (엔트리 이름 불변).

    ✓ python autobible.py + python -m bibleclip 둘 다 실행
  • Phase 5 · 빌드 검증

    로컬 PyInstaller 빌드 + 데이터 동봉 확인

    패키지 분할로 PyInstaller가 서브모듈을 놓치지 않는지 확인. 필요 시 spec에 hiddenimports=['bibleclip.ui.viewer', ...] 추가 또는 --collect-submodules bibleclip.

    ✓ dist 빌드 실행 + 자동 업데이트 흐름 점검
  • Phase 6 · 릴리스

    v1.6.0 태그 → CI 자동 빌드

    CHANGELOG에 "내부 구조 모듈화(기능 동일)" 기록 후 __version__=1.6.0 범프, 태그 푸시. Windows/macOS zip·dmg 생성 확인. 기존 사용자는 인앱 업데이트로 그대로 받음.

    ✓ CI 3 잡 green + 자산 3개 확인

7위험 요소 & 대비

위험영향대비
PyInstaller가 동적 서브모듈 누락빌드는 되나 실행 시 ImportError --collect-submodules bibleclip 또는 spec hiddenimports
resolve_data_dir의 __file__ 기준 경로 변동성경 DB 못 찾음 config로 옮길 때 base_dir 계산을 패키지 위치 기준으로 검증 (Phase 2 게이트)
순환 import (ui ↔ core)import 에러 의존 방향 단방향 고정: ui → core → data → config. 역참조 금지
믹스인 간 메서드 이름 충돌오버라이드 사고 모두 self._ 단일 네임스페이스 → 이름 그대로 유지하면 충돌 없음(이미 유일)
settings 파일명 변경 실수사용자 설정 초기화 autobible_settings.json 문자열 절대 변경 금지 (config 상수화)
롤백: 모든 작업은 refactor/modularize 브랜치에서. 문제 시 git switch main 한 번으로 즉시 원복. main은 항상 동작하는 v1.5.0 유지.
