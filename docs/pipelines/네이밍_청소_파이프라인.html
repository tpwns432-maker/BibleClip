  • 엔트리 파일명 → bibleclip 패키지/슈팅
  • tmpdir 접두사 bibleclip_update_

  • build_mac.sh · CI 빌드 참조

⚠ 마이그레이션 필요

바꾸면 기존 사용자 데이터가 사라짐. 이전 코드 필요.

  • autobible_settings.json
    → bibleclip_settings.json
  • 새 파일 없고 옛 파일 있으면 1회 자동 이전
  • 옛 파일은 한동안 폴백으로 읽음
  • .gitignore에 둘 다 등재

🔒 의도적 유지

  • GITHUB_REPO="AutoBible"
    (릴리스·자동업데이트 URL)
  • CHANGELOG의 과거 이름 언급
    (버전 이력은 사실 기록)
  • macOS 업데이터의 옛 번들 호환 로직
    ("AutoBible 2.app" 탐지)
가장 위험한 함정 — 설정 파일: autobible_settings.json을 그냥 이름만 바꾸면, 업데이트한 모든 기존 사용자의 버전 순서·마지막 위치·다크모드·창 크기가 전부 초기화됩니다. 반드시 마이그레이션 슈팅을 먼저 넣습니다.

3설정 마이그레이션 방식

새 이름으로 바꾸되, 옛 설정을 한 번 읽어와 그대로 이어쓰는 무손실 전환.

NEW_SETTINGS = "bibleclip_settings.json"
OLD_SETTINGS = "autobible_settings.json"   # 폴백용 (한동안 유지)

def _settings_path(self):
    return os.path.join(self._config_dir(), NEW_SETTINGS)

def _load_settings(self):
    new = self._settings_path()
    old = os.path.join(self._config_dir(), OLD_SETTINGS)
    # 새 파일이 없고 옛 파일이 있으면 → 1회 이전
    path = new if os.path.exists(new) else (old if os.path.exists(old) else new)
    try:
        with open(path, encoding="utf-8") as f:
            self.settings = json.load(f)
    except Exception:
        self.settings = {}
    # 다음 저장부터는 무조건 새 파일로 (옛 파일은 그대로 남겨둬도 무해)

def _save_settings(self):
    with open(self._settings_path(), "w", encoding="utf-8") as f:
        json.dump(self.settings, f, ensure_ascii=False, indent=2)

효과: 기존 사용자 → 첫 실행 시 옛 설정 자동 승계 후 새 파일로 저장. 신규 사용자 → 처음부터 새 파일. 옛 파일을 지우지 않으므로 구버전으로 되돌려도 안전(롤백 호환).

참고: 현재 코드의 설정 경로 처리 방식(루트의 autobible_settings.json)에 맞춰 위 슈팅을 이식합니다. _config_dir()는 기존 로직 그대로 사용 — 경로 정책은 바꾸지 않고 파일명만 전환.

4엔트리 파일명 전환

autobible.py는 spec·CI·build_mac.sh 3곳이 참조 → 같은 커밋에서 함께 바꿔야 빌드가 안 끊김.

파일변경 전 → 후
엔트리autobible.py → bibleclip.py (또는 -m bibleclip)
build.yml ×4--name BibleClip autobible.py → ... bibleclip.py
build.yml 버전추출open('autobible.py') → open('bibleclip.py')
build_mac.sh ×2autobible.py → bibleclip.py
*.spec['autobible.py'], name='AutoBible' → ['bibleclip.py'], name='BibleClip'
모듈화와 겹침: 모듈화 파이프라인을 먼저 하면 autobible.py는 이미 bibleclip/ 패키지를 부르는 3줄 슈팅이 됩니다. 그때 이 슈팅 파일명만 bibleclip.py로 바꾸거나 CI를 python -m bibleclip으로 전환하면 끝 — 중복 작업이 사라집니다.

네이밍 청소

슈팅 파일명·설정·내부 흔적 전환 + 마이그레이션. 엔트리명을 마지막에 한 번만 flip.

왜 이 순서인가

이사 ①잡파일을 먼저 비워야 모듈화 때 탐색·이동이 깔끔. 가장 안전해서 워밍업으로 적합.
모듈화 ②이 과정에서 autobible.py가 이미 슈팅이 됨 → 네이밍 청소의 "엔트리 파일명 전환"이 1줄 작업으로 축소.
청소 ③엔트리명·설정파일명을 가장 마지막에 바꾼다. 앞 두 작업이 이 이름들에 의존(안정)하기 때문. 한 번만 flip하면 끝.
